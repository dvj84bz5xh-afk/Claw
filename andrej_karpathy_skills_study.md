## 二、CLAUDE.md 四大核心准则

### 准则1：编码前先思考

**核心要求**：
- 明确陈述你的假设，如果不确定就提问
- 如果存在多种解读，全部列出来，不要默默选择其中一种
- 如果有更简单的方案，说出来，在合理情况下提出反对意见
- 如果内容不清晰，停止操作，明确说明困惑的点，再提问

**对您的意义**：
在诈骗调查数据分析中，这意味着 AI 不应默默假设数据格式，而应主动澄清：
- "这份 Excel 中的'金额'字段，是人民币还是美元？"
- "时间戳格式是 UTC 还是北京时间？"

---

### 准则2：优先简洁

**核心要求**：
- 不要添加需求之外的功能
- 单次使用的代码不要做抽象
- 不要添加未被要求的"灵活性"或"可配置性"
- 不要为不可能出现的场景做错误处理
- 如果你写了 200 行但可以用 50 行实现，就重写

**对您的意义**：
在编写诈骗调查工具时：
- 先用最简代码跑通流程，再考虑泛化
- 避免"过度工程化"——很多调查工具是一次性使用

---

### 准则3：精准修改

**核心要求**：
- 只修改必须修改的内容，只清理自己造成的混乱
- 不要"优化"相邻代码、注释或格式
- 不要重构没有问题的代码
- 匹配现有代码风格，即使你有不同的实现偏好
- 如果发现不相关的死代码，提出来，不要直接删除

**判断标准**：每一行修改的内容都应该能直接追溯到用户的请求。

**对您的意义**：
在维护现有诈骗调查代码库时：
- 修复 bug 时，只修改 bug 相关的代码
- 不要顺手"优化"其他函数——可能产生新 bug

---

### 准则4：目标驱动执行

**核心要求**：
把任务转化为可验证的目标：
- "添加校验" → "先为无效输入写测试用例，再让测试通过"
- "修复 bug" → "先写能复现 bug 的测试用例，再让测试通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务，列出简要计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

**对您的意义**：
在生成调查报告时：
- 先定义"什么是合格的报告"（格式、数据来源、结论结构）
- 再让 AI 生成，并对照检查项验证

## 三、如何在您的项目中应用

### 步骤1：在诈骗调查项目中创建 `CLAUDE.md`

创建文件：`c:\Users\10127\WorkBuddy\Claw\CLAUDE.md`

```markdown
# CLAUDE.md - 诈骗调查工具开发规范

## 项目背景
本项目用于柬埔寨波贝地区诈骗园区的调查数据分析，包括：
- TikTok 视频素材下载与整理
- 加密货币洗钱路径分析
- 调查报告自动生成

## 编码前先思考
- 所有数据字段必须明确单位（人民币/美元、北京时间/UTC）
- 如果数据来源不明确，先提问再编码
- 加密货币地址必须验证格式（BTC/ETH/TRX）

## 优先简洁
- 调查脚本优先跑通流程，不过度泛化
- 一次性分析脚本不需要抽象层
- 避免为"未来可能用到"的功能编写代码

## 精准修改
- 修复数据分析 bug 时，只修改相关代码
- 不要顺手"优化"不相关的分析函数
- 匹配现有代码风格（Python PEP8）

## 目标驱动执行
- 每个分析任务必须先定义验证标准
- 例如："识别可疑交易" → "先定义可疑标准（单笔>50万、频率>10次/日），再写检测逻辑"
- 多步骤分析必须列出验证检查点

## 数据规范
- 所有金额字段必须标注货币单位
- 所有时间字段必须标注时区
- 加密货币交易必须记录 txid 和确认数

## 输出规范
- 调查报告必须使用 Markdown 格式
- 所有数据来源必须标注获取时间
- 结论必须区分"已证实"和"推测"
```
